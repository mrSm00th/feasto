Engineering decisions

Why, not just what

Every non-obvious choice has a reason. These are the ones worth talking about.

📍
Why PostGIS over Redis GEO
Geo-proximity with PostGIS
ST_DWithin on a GIST-indexed Geography column runs in ~3ms on 10k rows. Redis GEO lacks JOIN support — you can't filter by cuisine or status in the same query.
📒
Why append-only
Immutable earnings ledger
Earnings records are never updated or deleted — only appended. Audit trail is always complete, payout reconciliation is always correct, no race conditions on financial writes.
⚡
Why cache-aside with soft failure
Redis cache strategy
Cache misses fall through to DB silently. A Redis outage degrades performance but never takes the app down. Discovery feed, restaurant detail, and cuisine lists are cached — never order or payment data.
🔄
Why triggers over app-layer
PostGIS location sync
DB triggers sync the Geography column from lat/lon on every write. The app never touches the Geography column directly — no risk of forgetting to update it in a new route.
⭐
Why one table
Polymorphic review model
A single Review table handles restaurant and rider reviews via nullable foreign keys. No join table, no type enum gymnastics. Queries stay simple, aggregations stay fast.
🔐
Why two buckets
Split S3 storage
Public bucket for restaurant and menu images — served at static URLs. Private bucket for KYC documents and rider identity proofs — only accessible via short-lived presigned URLs generated server-side.
Modules

What's inside

Eight domain modules, each with its own models, routes, schemas, and services.

01
Auth & Users
JWT access tokens, refresh token rotation, email OTP verification, password reset via secure emailed link. Role-based access: Customer, Owner, Rider, Admin.
02
Restaurants
Full onboarding state machine (DRAFT → ACTIVE), manual pause/resume, planned closures, PostGIS location intake, weekly availability schedules with multi-shift support.
03
Menus
Category and item CRUD with archive-before-delete enforcement. Drag-and-drop reordering, availability toggles, image upload with Pillow processing pipeline.
04
Orders & Cart
Full order lifecycle state machine (AWAITING_PAYMENT → DELIVERED). Cart with per-item quantity management. SELECT FOR UPDATE on checkout to prevent overselling.
05
Payments
Razorpay Orders API integration, webhook signature verification, COD support, automatic refunds on cancellation. Payment status tracked independently from order status.
06
Riders & Dispatch
Rider onboarding with PII encryption and private S3 KYC storage. PostGIS-based geo-dispatch to nearby riders. Real-time location updates, assignment timeout via Celery.
07
Discovery
Geo-proximity restaurant feed using ST_DWithin with cursor pagination. Cuisine filtering, city-based fallback. Cached at Redis with 30s TTL, invalidated on restaurant state changes.
08
Earnings & Payouts
Immutable earnings ledger per order. Weekly payout batching via Celery Beat. Commission rate configurable per restaurant. Separate rider and restaurant payout flows.
