Feasto is a food delivery backend I built to work through the kinds of problems that show up in real systems: location search, payments, caching, background jobs, and real-time updates.

Solo project. I wanted to actually work through system design problems instead of just reading about them, so I picked something with a lot going on — geo queries, caching, payments, real-time updates — and wrote up the tradeoffs and bugs below as I ran into them.

Engineering decisions

Why, not just what

A few decisions that took more than one attempt to get right, and the reasoning behind where I landed.

📍
Why PostGIS, not day one
From Haversine to PostGIS
Rider matching started with plain latitude/longitude columns: a SQL bounding-box pre-filter narrowed the candidates, then Haversine ran in Python on what was left. It worked, but couldn't use an index. Rider and restaurant locations moved to a PostGIS Geography column with ST_DWithin on a GIST index — distance and relational filtering now happen in one indexed query.
📒
Why append-only
Immutable earnings ledger
Earnings records are never updated or deleted, only appended, then later marked PAID_OUT in a batch. So there's always a full history to check, and I don't have to worry about two writes stepping on the same balance at once.
🔒
Why SELECT FOR UPDATE
Race condition prevention
Two riders accepting the same order at the same time was a real bug I ran into, not something I planned around ahead of time. SELECT FOR UPDATE on the order row fixes it — the second request just fails instead of double-assigning the order.
🔄
Why triggers over app-layer
PostGIS location sync
A DB trigger syncs the Geography column from lat/lon on every write. The app never touches that column directly, so I can't forget to sync it from a new route or migration.
⭐
Why one table
Polymorphic review model
One Review table handles both restaurant and rider reviews via nullable foreign keys, instead of a separate join table or a type enum. Didn't seem worth the extra complexity for something this simple, and the queries stayed easy to write.
🔐
Why two buckets
Split S3 storage
Public bucket for restaurant and menu images, served directly. Private bucket for KYC documents and rider ID proofs — those only go out through short-lived presigned URLs generated server-side.
Redis caching

Caching restaurant reads with Redis

Cache-aside pattern on the high-read endpoints. Redis is a soft dependency — if it goes down, requests fall back to Postgres and get a bit slower, but nothing breaks.

Modules

Core platform modules

15 domain modules, 117 endpoints, grouped into 8 areas below — each with its own models, routes, schemas, and services.

01
Auth & Users
JWT access tokens, refresh token rotation, email OTP verification, password reset via emailed link. Role-based access: Customer, Owner, Rider, Admin. Case-insensitive email dedup.
02
Restaurants
Onboarding state machine (DRAFT → ACTIVE), manual pause/resume, planned closures, PostGIS location intake, multi-shift weekly availability, cuisine tagging with admin review.
03
Menus
Category and item CRUD with archive-before-delete enforcement. Drag-and-drop reordering, availability toggles, image upload through a Pillow pipeline for JPEG conversion and thumbnails.
04
Orders & Cart
Order lifecycle state machine (AWAITING_PAYMENT → DELIVERED). Cart with per-item quantity management.
05
Payments
Razorpay Orders API integration, signature-verified webhooks, COD support, automatic refunds on cancellation. Payment status tracked separately from order status.
06
Riders & Dispatch
Rider onboarding with PII encryption and private S3 KYC storage. PostGIS-based dispatch searches a radius, falls back to a wider radius if empty, and notifies a pool of nearby riders who race to accept.
07
Discovery
Geo-proximity restaurant feed using ST_DWithin with cursor pagination. Cuisine filtering, city-based fallback. Cached in Redis with a 30s TTL, invalidated on restaurant state change or pause.
08
Earnings & Payouts
Append-only earnings ledger per order. Weekly payout batching via Celery Beat. Commission rate configurable per restaurant. Separate rider and restaurant payout flows with an audit trail.

Lessons learned

Problems I ran into while building this

A few things that broke, or would have broken, before I caught them — not patterns I planned up front, just what came up.

🔏
Why verify, then guard
Trusting the webhook
Razorpay webhooks arrive as unauthenticated POST requests, so the signature gets HMAC-verified against the Razorpay secret before anything is parsed. Razorpay also retries webhooks on slow responses — each handler checks the payment's current status before applying an update, so a retried delivery can't double-process the same payment.
🖼️
Why a two-phase upload
Image pipeline reliability
First version wrote the DB record before the S3 upload finished, so a failed upload left a menu item pointing at an image that didn't exist. Now the upload has to complete and get verified first — the DB row only gets written once the file is actually in storage.
🛵
Why a radius fallback
No riders found nearby
Dispatch first searches for riders within the standard radius. If none are found, it retries once at a wider max radius before giving up. Whichever riders turn up get notified as a pool — they race to accept, and the first POST /accept wins.
📧
Why I switched providers twice
Email deliverability
Transactional email started on Resend, which only allows sandbox sending without domain verification. Moved to Gmail SMTP for a working free option, but delivery landed in spam at real inboxes. Settled on Brevo SMTP via aiosmtplib, which has handled verification and reset emails reliably since.
